JDK源码之util

发表于 2020-03-14 | 更新于: 2020-04-06 | 分类于 源码分析 |

以下源码均来自于jdk8。

从Abstract类看集合


除了AbstractMap直接继承于Map接口，其余Abstract类构成了如上的类继承图。

Iterable

可以看到，Iterable接口位于最顶层，其声明的方法有：

  • Iterator<T> iterator()
  • void forEach(Consumer<? super T> action)
  • Spliterator<T> spliterator()

forEach方法主要是为了适应jdk8函数式编程；spliterator方法则是利用多线程并发处理，从而加快遍历速度。但是，我们今天着重要讲的是Iterator接口。

public interface Iterator<E> {
    boolean hasNext();
    E next();
    void remove();
    void forEachRemaining(Consumer<? super E> action);
}

Iterator接口可以说是所有集合的一个核心部分，往大了说就是采用了迭代器模式，提供给用户一种简单的遍历方法，但却不暴露内部的实现细节。具体来讲就是，我们能够用一种简单且通用的方法来遍历集合内部的元素。

/********   1   ********/
Iterator<Integer> it = list.iterator();
while (it.hasNext()) {
    System.out.println(it.next());
    // it.remove()
}

/********   2   ********/
for (int i: list) {
    System.out.println(i);
}

第一种做法是显式调用Iterator方法进行遍历，第二种则是java提供的语法糖，但实质也是将其转化为Iterator方法再进行执行。

Collection

Collection主要提供了一般集合需要提供的方法，包括是否为空，添加新元素，删除元素、转为数组等。

AbstractCollection

AbstractCollection是所有后续AbstractList、AbstractSet等的父类，其大多数方法主要是通过迭代器实现的。源码有几个有趣的地方。

public boolean addAll(Collection<? extends E> c) {
    boolean modified = false;
    for (E e : c)
        if (add(e))
            modified = true;
    return modified;
}

第一个有趣之处在于只要正确添加第一个元素，那么返回的永远是true。
第二个则是源码中MAX_ARRAY_SIZE=Integer.MAX_VALUE - 8，并且注释里写了这样设置是因为有些JVM会在数组里保存头部信息。hugeCapacity方法用于在将集合转数组时，若数组容量不够且满足一定条件时就会被调用。

private static int hugeCapacity(int minCapacity) {
    if (minCapacity < 0) // overflow
        throw new OutOfMemoryError
            ("Required array size too large");
    return (minCapacity > MAX_ARRAY_SIZE) ?
        Integer.MAX_VALUE :
        MAX_ARRAY_SIZE;
}

但是其源码中却出现了返回Integer.MAX_VALUE的可能。个人对此的猜想是，上层方法调用hugeCapacity方法时，minCapacity参数不会大于MAX_ARRAY_SIZE。

AbstractList

AbstractSet和AbstractQueue的源码比较简单，没多少东西，所以我们只讲AbstractList。AbstractList提供了一种Iterator的实现Itr（除此之外，还有一种ListIterator的实现ListItr），Itr里有一个成员变量expectedModCount，与AbstractList的成员变量modCount相互关联。具体的做法是：

  1. 在创建Itr对象时，会将该对象的expectedModCount置为和modCount一致。
  2. 每次调用Itr对象的next和remove方法时，都会检查expectedModCount是否等于modCount：
    1. 若相等，则表明这段时间AbstractList对象未被修改，则可以正确执行
    2. 若不相等，则抛出运行时异常
  3. 对于使用Itr对象进行遍历时，只有Itr对象的remove方法才是合法的。

这种机制保证了使用Iterator遍历List时不能修改List。

AbstractMap

AbstraceMap实现Map接口，主要提供了两种Map.Entry的实现：

  • SimpleEntry：可变Entry
  • SimpleImmutableEntry：不可变Entry

具体的集合实现

接下来介绍几个具体集合的实现细节。

ArrayList

其内部存储结构其实就是elementData数组，具体的实现细节有：

  • 未指定初始容量，那么elementData的初始容量为0，在第一次进行add后容量变为10。
  • 在数组扩容时，每次以1.5倍扩张

HashMap

HashMap的内部存储结构是哈希桶数组，其具体细节有：

  • 初始的加载因子为0.75，哈希桶数量为16。
  • 不直接使用key的hashcode，而是对hashcode进一步处理，得到真正可用的hash值（个人其实不太理解为何要进行这一步处理，因为hash应该取决于元素本身的hashcode方法，查了一些资料，说是尽量让高低位都参与运算，可能对于后续查找数组中的哈希桶有帮助）
  • 每次调用put方法时，都会先找到对应的哈希桶（计算得到的hash值&哈希桶数量-1），然后遍历桶上的链表，看key是否已存在：
    • 若key存在，则替换该key的value值
    • 若key不存在，则进行尾插，但是若该链表元素数量超过树化限制（8），则会将该链表转化为红黑树。
  • 当HashMap的元素数量大于threshold（加载因子*hash桶数量）时，并且重新调整内部node的位置。

HashSet

其实质就是HashMap，只是Value部分全都由一个内置的Object对象替代。
